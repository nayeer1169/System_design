Slide 1: What is System Design?

Architectural Fundamentals
  • Scope: Defining components, modules, interfaces, and databases for massive scale.
  • Key Drivers: High traffic volumes, massive data storage, and low latency needs.
  • The Golden Rule: System design is an exercise in balancing trade-offs (Speed vs. Consistency vs. Resource Cost).

Slide 2: Scalability: Scale Up vs. Scale Out

Resource Allocation
  • Vertical Scaling (Scale Up): Upgrading CPU/RAM. Simple but hits physical boundaries and introduces a Single Point of Failure (SPOF).
  • Horizontal Scaling (Scale Out): Adding more server nodes. Operates on commodity hardware and enables high fault tolerance.
  • Modern Practice: Horizontal scale is standard for cloud-native web systems.

Slide 3: Load Balancing & Traffic Control

API Gateway & Proxying
  • Nginx/HAProxy: Sits as a reverse proxy in front of server pools to distribute requests.
  • Health Checks: Automatically ping servers to detect failures and route requests only to healthy nodes.
  • Algorithms: Round Robin (sequential distribution) and IP Hash (maintaining sticky sessions).

Slide 4: Caching Layer (RAM Cache)

Performance Optimization
  • RAM Access: Storing key-value pairs in memory (Redis/Memcached) is 100x faster than disk access.
  • Cache Hit/Miss: App checks cache first (Hit). If not found (Miss), queries the DB and writes back to cache.
  • Trade-off: Cache eviction policies (e.g., LRU, LFU) must be managed to handle stale data.

Slide 5: Relational (SQL) vs. Non-Relational (NoSQL)

Database Storage
  • SQL (e.g., PostgreSQL): Predefined schemas, strict ACID compliance. Best for structured, relational transactions.
  • NoSQL (e.g., Cassandra, MongoDB): Schema-less, document or key-value layouts. Built for horizontal scaling.
  • Strategy: Choose SQL for strong safety, and NoSQL for rapid scaling or semi-structured data.

Slide 6: The CAP Theorem

Distributed Consistency
  • Consistency (C): Every node returns the latest write or an error.
  • Availability (A): Every non-failing node returns a response without error guarantees.
  • Partition Tolerance (P): System runs despite dropped network messages. You must choose CP or AP.
